UI(UserInterface)
- 제품이나 소프트웨어와 상호작용할 때 직접 보거나 조작하는 화면 요소
- 사용자가 눈으로 보거나 손으로 누르는 모든 구성요소
UX(UserExperience)
- 사용자가 소프트웨어를 사용하면서 느끼는 전체적인 경험
- 서비스를 이용하면서 느끼는 편리함, 만족감, 효율성, 즐거움 등을 모두 포함한다.

UI 구성 요소
1. 시각적 디자인(Visual Design)
- 사용자가 화면에서 보는 모든 요소
- 색상, 글꼴, 아이콘, 이미지, 레이아웃
2. 상호작용 디자인(Interaction Design)
- 사용자의 행동에 대해 시스템이 어떻게 반응하는지 설계
- 좋은 상호작용 디자인은 사용자가 어떤 상태인지 쉽게 알 수 있도록 도와준다.
3. 정보구조(Information Architecture)
- 정보를 체계적으로 분류하고 배치하는 방법
- 사용자가 원하는 정보를 쉽게 찾을 수 있도록 구성하는 것

UX 구성 요소
1. 사용성(Usability)
- 사용자가 얼마나 서비스를 쉽게 이용할 수 있는가
2. 유용성(Usefulness)
- 서비스가 사용자의 요구사항을 얼마나 잘 충족하는지 의미한다.
3. 매력도(Desirability)
- 서비스가 사용자에게 얼마나 매력적으로 느껴지는가
4. 접근성(Accessibility)
- 모든 사용자가 서비스를 이용할 수 있도록 만드는 정도
- 시각장애인, 청각장애인, 고령자, 색약 사용자 등도 서비스를 사용할 수 있어야 한다.
5. 신뢰성(Credibility)
- 사용자가 서비스와 정보를 얼마나 믿을 수 있는가

UI/UX의 관계
- 좋은 UI는 좋은 UX에 도움이 된다.
- 항상 좋은 UI가 좋은 UX는 아니다.

주요내용 (사용자 중심 설계를 위한 6가지 원칙)
1. 사용자는 설계 과정 전반에 적극적으로 참여한다.
- 실제 사용자의 의견을 먼저 수집한다.
2. 사용자 중심 설계는 전체 사용자 경험을 고려해야 한다.
- 기능만 동작한다고 되는게 아니라 사용자가 느끼는 감정도 고려
3. 설계는 반복적으로 이루어져야 한다.
- 계속 테스트를 하고 수정해서 UI를 만들어나가야한다.
4. 설계는 사용자와 소프트웨어 구축 목표 및 환경을 기반으로 한다.
- 누가, 어떤 환경에서 사용하는지 고려해야 한다.
5. 사용자 중심 설계 팀은 여러 학문 분야가 함께 협력해야 한다.
- UI/UX는 한사람이 만들 수 없다.
- 여러 분야의 사람이 함께 참여해야 한다.

UI 디자이너 : 화면 디자인

개발자 : 기능 구현
관리자 : 업무 요구사항 제공
6. 사용자 평가에 반드시 포함되어야 한다.
- 실제 사용자가 평가를 하고 개선해야 한다.

카드 레이아웃을 분석하고 특성을 정리한다.
ㄴ 정보 구분이 쉽다.
ㄴ 모바일 환경에 적합하다.(손가락으로 누르기 쉽고, 하나의 콘텐츠를 카드 단위로 구분하기가 좋다.)
ㄴ 확장성이 좋다.
ㄴ 많은 정보를 한 번에 보기 어렵다
ㄴ 공간을 많이 사용한다.(카드 하나가 차지하는 높이가 커서 한 화면에 여러 정보를 보여주기가 어렵다)

모달(Modal) 사례를 분석하고 특성을 정리한다.
ㄴ 사용자의 집중을 유도한다.
ㄴ 중요한 정보를 강조할 수 있다.
ㄴ 너무 많이 사용하면 불편하다.
ㄴ 작은 화면에서는 공간이 부족하다.

4-3. 사용자 편의성과 UX 요소 비교 평가를 한다.
- 단순히 UI를 분석하는 것만으로는 부족하다.
- 사용자 입장에서 얼마나 편리한지 평가해야 한다.
평가 항목
사용성 : 쉽게 사용할 수 있는가
유용성 : 필요한 기능을 제공하는가
접근성 : 누구나 사용할 수 있는가
일관성 : 같은 규칙을 유지하는가
만족도 : 사용자가 만족하는가

4-4. 사례 분석을 통해 UI 설계 트렌드와 패턴을 도출한다.
트렌드 : 현재 많은 서비스가 공통적으로 사용하는 설계 방식
패턴 : 반복적으로 사용되는 UI 설계 방법
- 최근 트렌드
다크모드 지원
카드 레이아웃 사용
모바일 우선 설계
간결한 메뉴 구조
직관적인 아이콘 사용

5. 목표 응용소프트웨어의 UI요구사항을 확인한다.
5-1. 목표 응용소프트웨어의 기능 및 사용자를 분석한다.
- 소프트웨어 구축 목적 및 주요 기능 문서를 검토한다.
검토 대상 문서
기획서 : 프로젝트 목적
제안서 : 구축 필요성
회의록 : 요구사항 논의 내용
요구사항 명세서 : 구현할 기능 목록
업무 프로세스 문서 : 현재 업무 흐름

5-2. 목표 응용소프트웨어의 기능 목록을 작성하고 이를 유형으로 분류
- UI는 기능을 제공하기 위한 수단
- 따라서 어떤 기능이 있는지 먼저 정리해야 한다.
예시
기능 목록 작성
기능명	설명
로그인	사용자 인증
출결 등록	출석 정보 입력
출결 조회	출결 현황 조회
상담 등록	상담 내용 작성
상담 조회	상담 이력 조회
수강료 조회	납부 내역 확인
통계 조회	월별 통계 확인

기능은 성격에 따라 분류할 수 있다.
기능 유형 분류
유형		기능예시
입력 기능	회원가입, 출결 등록
조회 기능	목록 조회, 검색
수정 기능	정보 변경
삭제 기능	데이터 삭제
통계 기능	그래프, 차트

분류하는 이유
- 기능 유형에 따라 화면 설계 방식이 달라지기 때문이다.

입력기능 -> 입력창 필요
조회기능 -> 목록 화면 필요
통계 기능 -> 차트 필요

5-3. 목표 응용 소프트웨어의 사용자를 식별하여 정의한다.
- 같은 기능이라도 사용자에 따라 필요한 화면이 다르다.
- 사용자 특성에 따라 UI 복잡도가 달라진다.
IT 활용 능력이 낮음 -> 단순한 화면 필요
IT 활용 능력이 높음 -> 다양한 기능 제공이 가능하다.

5-4. 사용자 유형별 사용 목적 및 기대 행동을 정의한다.
강사
사용목적
- 출결 입력
- 출결 확인

기대행동
출결을 입력한다.
학생 목록을 조회한다.

관리자
사용 목적
- 매출 확인
- 수강생 현황 확인

기대행동
- 통계 조회
- 보고서 출력

목적을 이해하면 필요한 메뉴와 화면을 결정할 수 있다.

5-5. 사용자 기기, 운영체제, 이용 시간대 등 사용자 환경을 분석한다.
사용자 환경 : 사용자가 서비스를 사용하는 환경
분석항목
기기(Device) : PC,노트북, 태블릿, 스마트폰
운영체제(OS) : Windows, macOS, Android, iOS
이용장소 : 사무실,학교,집,야외
이용시간대 : 업무 시간, 야간, 주말

예시
강사
기기 : PC
운영체제 : Windows
시간대 : 수업 전후

5-6. 사용자 경험상의 주요 고려사항을 도출한다.
- 사용자가 서비스를 사용할 때 느끼는 편의성과 만족도를 고려

클릭수 최소화 : 목표까지 도달하기 위해 많은 클릭은 피곤하다
명확한 피드백 제공 : 클릭해도 반응이 없으면 저장 여부를 알 수 없다.
색상대비 확보 : 배경과 글씨가 구분이 잘 되도록 한다.
일관성 유지 : 저장 -> 파란색, 삭제 -> 빨간색 모든 화면이 동일해야 함
접근성 고려 : 대체 텍스트, 키보드 사용, 색약 모드 제공

5-7. 기능별 사용자 사용 시나리오 작성하기
- 사용자가 어떤 순서로 화면을 사용하게 되는가
- 사용 시나리오를 작성하면 어떤 화면이 필요한지, 버튼이 필요한지, 어떤 순서로 이동하는지 쉽게 파악할 수 있다.

6. 업무의 특성을 고려하여 UI 구성 요소 및 흐름 파악하기
6-1. 화면별 주요 구성요소인 버튼, 폼, 메뉴등의 목록 작성하기
- 각 구성 요소의 명칭, 기능, 위치 등을 정리하여 응용소프트웨어의 각 화면에 포함된 UI구성 요소를 식별하고 목록화한다.

6-2. 화면간 이동경로인 UI 흐름도를 작성한다.
- 사용자가 화면과 화면 사이를 어떻게 이동하는지 표현한 그림
고려해야 하는 사항
- 사용자 이동 경로 : 사용자가 원하는 기능까지 몇 번 클릭하는지
- 화면 간 관계 : 현재 화면 -> 다음 화면
- 예외 상항 : 로그인 실패 -> 로그인 화면 유지

UI 흐름도를 작성하는 이유
- 화면 누락 방지
- 기능 누락 방지
- 개발 범위 파악
- 사용자 경험 개선

6-3. 핵심 기능인 상호작용 방식을 분석한다.
- 상호작용 : 사용자가 소프트웨어를 조작했을 때 시스템이 반응하는 과정
- 같은 기능이라도 상호작용 방식에 따라 사용자의 편의성이 크게 달라진다.

6-4. 디자인 일관성, 접근성, 사용성을 평가한다.
- 전체 UI에서 색상, 폰트, 버튼 형태 간격 등이 일관성 있게 적용되었는지 확인하고 장애인나 고령자 등도 쉽게 사용할 수 있는지 확인

6-5. 경쟁 소프트웨어의 UI 구성을 비교 분석
- 경쟁 서비스의 장점을 참고하고 단점을 개선하기 위해서
분석 항목
내비게이션 메뉴 : 사용자가 이동하는 방식
버튼 디자인 : 버튼 크기, 색상, 형태
입력 폼 : 사용자가 데이터를 입력하는 방식
사용자 흐름 : 작업 수행 과정

7. 사용자 면담 및 UI/UX 요구사항 수집하기
- 개발자나 디자이너가 생각하는 좋은 화면과 실제 사용자가 원하는 화면은 다를 수 있다.

7-1. 사용자 면담 또는 설문 대상자를 선정
- 시스템 구축 목표와 관련된 주요 사용자 그룹을 정의
- 사용자 그룹별 특성과 사용 목적을 파악
- 설문 대상자의 표본 크기와 선정 기준을 확정
- 설문 참여자 모집 및 일정 조율(이메일,문자,전화,사내 공지)

7-2. 면담 질문지 또는 설문지를 작성하고 계획을 수립한다.
- 면담/설문 목적과 목표를 명확하게 정의한다.
목적예시 : 현재 시스템의 불편사항 파악, UI개선 요구사항, 사용자 만족도
목표예시 : 사용자 만족도 80% 이상 달성

- 질문의 초안을 작성한다
기본정보 : 연령대, 사용기기, 사용 경험
현재사용경험 : 현재 UI 만족도, 자주 사용하는 기능, 불편한 화면
UI 개선 요구사항 : 버튼크기, 폰트크기, 메뉴이동 편의성
전반적인 개선 방향 : 가장 먼저 개선해야할 사항

- 질문의 유형을 정의한다.
객관식
주관식
혼합형 질문

- 설문지/면담 질의서 템플릿을 작성한다.
- 면담/설문 계획을 수립한다.

7-3. 면담/설문을 사전에 검증하여 수정 보완한다.
- 내부 검토를 통해 질문 내용의 적절성을 평가한다.
- 소수의 사용자에게 파일럿 면담/설문을 진행한다.
ㄴ 파일럿 조사 : 실제 조사 전 소규모로 먼저 테스트 하는것
- 이해하기 어려운 질문, 중복 항목, 누락된 질문을 파악

7-4. 사용자 대상 면담 및 설문을 진행한다.
- 조사 목적과 일정을 공지한다.
- 정의한 면담 방식에 따라 면담을 수행한다.
- 설문을 진행할 경우 온라인 링크 또는 오프라인으로 설문지를 배포한다.
- 면담 및 설문 응답 결과를 취합
- 면담 및 설문 참여자 보상과 결과에 대한 피드백을 공유한다.

7-5. 기능 요구사항과 UI 개선 요구사항을 구분한다.
- 수집된 응답 내용을 기능/디자인으로 분류한다.
- 기능 개선 요청과 UI 개선 요청으로 태깅한다.
- 분류 기준과 에시를 작성한 후 데이터를 정리한다.

7-6. 유사 서비스 등에서 발생한 반복적 불만/요구사항의 우선순위를 도출한다.
- 경쟁사 및 유사 서비스의 의견을 조사한다.
- 반복적으로 나타나는 불만 사항을 정리한다.
- 발생 빈도와 영향도에 다른 우선순위를 문서로 작성한다.
- 우선순위에 따른 개선 아이디어를 도출하고 보고서를 작성한다.

8. UI 요구사항을 도출하고 정리한다.
8-1. 수집된 요청 사항과 유사 서비스 등에서 발생한 불만 사항 등을 분석하여 결과를 통합한다.
- 사용자 설문, 면담, 로그 데이터 등에서 수집된 요청 사항을 목록화한다.
- 유사 서비스 리뷰 및 피드백 데이터를 수집한다.
- 중복 유사 요청 사항을 그룹핑 하여 항목을 정리한다.
- 분석 결과를 통합 보고서로 작성한다.
8-2. 사용 목적과 시나리오별 UI 요구사항을 도출한다.
- 사용자 그룹별 사용 목적을 분석한다.
- 대표 사용 시나리오를 작성한다
- 시나리오별 필요한 UI 구성 요소를 도출한다.
- 기능 흐름을 분석하여 UI 요구사항을 식별한다.
8-3. 기능 요구사항과 비기능요구사항을 분류한다.
8-4. UI 요구사항에 고유 ID를 부여하고 설명을 작성한다.
8-5. 요구사항의 우선순위와 난이도, 중요도 기준을 정의한다.
8-6. UI 요구사항명세서를 작성한다.
