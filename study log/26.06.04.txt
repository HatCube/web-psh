SQL 처리 흐름
구문분석
- SQL문이 사용된 적이 있는지 없는지 판단(공유 풀 영역을 검색하여 판단)
※공유 풀 영역
- 데이터베이스 메모리 안에 있는 공간
- 이전에 실행된 SQL문과 실행 계획이 저장될 수 있다.

실행이 된 적이 있음 -> 구문분석 할 필요 없음
실행 된 적 없음 -> 구문분석 작업 수행

구분 분석이란?
- 사용자가 작성한 SQL문장을 DBMS가 이해할 수 있는 형태로 해석하는 과정
SELECT * FROM EMP WHERE EMPNO = 7369;
DBMS는 이 문자열을 보고
SELECT문인지
문법이 올바른지
EMP 테이블이 존재하는지
EMPNO 컬럼이 존재하는지
사용자가 권한이 있는지
같은 내용을 확인해야 한다.

구문 분석의 3단계
1. 문법 검사 : SQL문법이 맞는지 확인
2. 의미 검사 : 문법은 맞지만 실제 객체가 존재하는지 확인
3. 권한 검사 : 사용자가 해당 객체에 접근 가능한지 확인


실행단계
- 구문 분석이 끝난 SQL을 실제로 실행하는 단계
- 구문 분석 단계에서 실행 계획이 만들어졌다면, 데이터베이스는 그 계획에 따라 실제 데이터를 찾거나 변경을한다.

※실행 계획
- 어떤방법으로 SQL을 실행하는게 가장 빠를까? 를 계산
- 옵티마이저 : 계산하고 실행하는 역할, 옵티마이저가 선택한 실행 방법이 실행 계획이다.

- 구문 분석이 정상적으로 끝나면 서버 스포세스는 메모리 영역의 데이터베이스 버퍼 캐시를 검사

※ 데이터베이스 버퍼 캐시
- 디스크에서 읽어온 데이터를 잠시 저장해두는 메모리 공간
- 디스크는 데이터를 영구적으로 저장하지만 속도가 느리기 때문에
- 메모리에 잠깐 저장해놓는다.

데이터베이스 버퍼캐시에 존재 -> 꺼내서 사용

없음 -> 데이터 파일에서 읽고 -> 버퍼캐시에 저장

SELECT가 아닌 DML이라면 버퍼 캐시 영역에서 수정, 삭제, 추가를 한다.
※ COMMIT을 해야 변경사항이 확정된다.

추출단계

- 실행 결과를 사용자(클라이언트)에게 보내주는 단계

SQL 작성 시 고려사항
1. SQL의 특성을 충분히 이해 해야 한다.(DBMS별 작성 방법)
2. 옵티마이저가 실행 계획을 수립한 후 실행된다는 것을 이해해야 한다.
- 이 조건은 인덱스를 사용할 수 있는가?
- 이 조인은 어떤 테이블부터 읽는 것이 좋은가?
- WHERE의 조건이 너무 넓지는 않은가?
- 정렬이 불필요하게 발생하지는 않는가?
3. 실행 계획에 따라 수행 속도 차이가 크게 발생할 수 있다는것을 이해해야 한다.
4. 옵티마이저가 비정상적으로 동작한다면, 이를 추적하여 개발자가 원하는 실행 계획으로 동작될 수 있도록 조정하는 과정이 필요하다.
- 인덱스 추가
- SQL의 조건을 변경
- JOIN의 순서를 개선
- 힌트

- 조건을 명확하게 작성해야 한다.

6. 좋은 SQL은 결과를 추론하고 집합적으로 접근해야 한다.

도메인

논리 데이터 모델의 구성 요소 중, 데이터베이스에 명시된 규정에 따라 릴레이션이 가질 수 있는 유효한 데이터 값의 범위를 제약하는 규칙을 의미하는 용어를 작성하시오. (예: 개체 무결성, 참조 무결성 등)

제약조건

관계 데이터베이스 모델에서 상위 테이블의 기본키(Primary Key)를 참조하여 테이블 간의 연결 고리 역할을 하는 속성을 무엇이라고 하는지 작성하시오.

외래키(Foreign key)

논리 데이터 모델링의 주요 목적 중 하나는 데이터의 중복을 제거하여 이상 현상(Anomaly)을 방지하는 것입니다.
데이터베이스에서 발생할 수 있는 이상 현상(Anomaly)의 3가지 종류를 제시하고, 각각의 개념을 간략히 설명하시오.

삽입이상
- 데이터를 추가할 때 필요없는 데이터도 같이 추가
삭제이상
- 데이터를 삭제할 때 필요한 데이터도 같이 삭제
갱신이상
- 일부 정보 수정시 데이터의 불일치가 발생하는 현상

데이터베이스 설계 과정에서 '개체 무결성(Entity Integrity)'과 '참조 무결성(Referential Integrity)'은 데이터의 정확성을 보장하는 핵심 규칙입니다. 
이 두 제약조건의 개념과 위배되지 않기 위한 세부 규칙을 비교하여 서술하시오.

개체 무결성
- NULL값이 들어가면 안되고, 중복되는 값이 들어가면 안된다.
참조 무결성
- 외래키 값은 참조 대상이 되는 관계 릴레이션의 기본키 값과 동일하거나, 혹은 NULL이어야 한다.

논리 데이터 저장소 확인 단계에서는 관계형 데이터 모델의 설계 품질을 높이기 위해 '정규화(Normalization)' 과정을 검토합니다. 
제1정규형(1NF)부터 제3정규형(3NF)까지의 정규화 과정을 각 단계별 변환 조건(만족해야 하는 기준)을 포함하여 단계적으로 서술하시오.

1정규형(1NF) : 모든 테이블의 값이 원자값이어야 한다.
2정규형(2NF) : 1정규형을 만족하면서, 기본키가 아닌 모든 속성이 기본키에 완전 함수적 종속을 만족해야 한다.
3정규형(3NF) : 2차 정규형을 만족하고, 기본키가 아닌 모든 속성이 기본키에 이행함수적 종속을 만족하지 않아야 한다.


관계형 데이터 모델에서 릴레이션의 행(Row)을 뜻하는 용어로, 카디널리티(Cardinality)의 수와 직결되는 이것은 무엇인가?

튜플

어떤 릴레이션 R에서 속성 X의 값 각각에 대해 속성 Y의 값이 오직 하나만 연관되어 있을 때, Y는 X에 '이것' 되어 있다고 한다. 무엇인가?

함수적 종속성

개념적 데이터 모델링 단계에서 
도출된 ERD를 관계형 데이터베이스의 논리적 테이블 구조로 변환하는 과정을 
'릴레이션 스키마 변환(Mapping Rule)'이라고 합니다. 
이 과정에서 1:N(일대다) 관계를 가진 두 개체가 
테이블로 변환될 때 외래키(Foreign Key)가 지정되는 규칙을 서술하시오.

1:N 관계에서 외래키는 항상 N쪽에 해당하는 자식 테이블에 생성된다.

1쪽 테이블의 기본키(PK)값을 가져와서 N쪽 테이블의 외래키 속성으로 추가하여 두 테이블 간의 연결고리를 형성한다.

데이터베이스의 정규화(Normalization) 과정은 데이터 무결성을 높여주지만, 과도한 정규화는 시스템의 성능 저하를 유발할 수 있습니다. 
정규화가 조회(Select) 성능을 떨어뜨리는 이유를 연산(Operation)의 관점에서 서술하시오.

시스템이 갑자기 느려졌을 때, 서버 컴퓨터 자원을 모니터링하는 '리소스 모니터링 도구'와 사용자의 거래를 끝까지 추적하는 '엔드투엔드(E2E) 모니터링 도구'의 결정적인 진단 차이점을 비교 서술하시오.

리소스 모니터링 도구는 서버 자체의 하드웨어만 감시하므로 시스템 상태가 '모두 정상'인 것처럼 보여 문제의 원인을 잡지 못한다.

반면 E2E 모니터링 도구는 사용자의 요청부터 내부서버, DB 조회, 외부 API 호출까지의 전 트랜잭션 과정을 결합 추적하므로 실제 장애 원인을 명확히 찾아낼 수 있다.

테스트 결과와 정의된 요구사항을 기준으로 데이터 조작 프로시저의 성능 이슈를 해결하기 위한 '성능 개선 및 해결 단계(6가지 핵심 방안)'를 순서대로 나열하시오.

속도를 저하시키는 SQL 구문 재작성
실행 계획 개선
인덱스 재구성
시스템 리소스 병목 구간 조정
개선 후 동일한 테스트로 효과 검증
성능 개선 결과 문서화 및 공유

데이터 조작 프로시저를 최적화할 때, 조건절 우변에 리터럴 값(예: '01012345678')을 직접 하드코딩하지 않고 WHERE phone = :b1과 같이 작성하는 방식을 무엇이라 하는가? 이 방식을 적용하면 SQL이 최초 1회만 파싱(Parsing)되어 공유 풀(Shared Pool)에 저장되므로 시스템 전체의 CPU 자원을 크게 절약할 수 있다.

바인드 변수

데이터베이스 성능 모니터링 도구 중, 시스템 전반의 '자원 사용량'을 중심으로 감시하는 인프라 모니터링과 대비되는 개념을 작성하시오. 애플리케이션의 사용자 요청(Transaction) 시작부터 DB 질의 수행 및 응답까지의 전체 흐름을 엔드투엔드(E2E)로 추적하고 병목 구간을 시각화해 주는 소프트웨어 성능 관리 솔루션의 약어입니다.

APM

절차형 SQL에서 실행 도중 발생할 수 있는 데이터 정합성 오류나 비즈니스 규칙 위배 상황을 처리하기 위해, 시스템 에러(예: Zero Divide)가 아님에도 개발자가 직접 정의하여 발생시키는 예외 처리 방식의 명칭을 작성하시오.

사용자 정의 예외

명시된 시스템의 안정적 운영 지표 중, "사용자가 필요할 때 시스템을 정상적으로 사용할 수 있는 정도"를 뜻하는 용어와, "시스템이 장애나 점검 등으로 인해 정상적으로 서비스를 제공하지 못하는 시간"을 뜻하는 핵심 용어 2가지를 차례대로 작성하시오.

가용성, 다운타임

사용자가 쿼리를 날렸을 때, '구문 분석(Parsing) 단계'에서 '공유 풀(Shared Pool)'이 수행하는 핵심 역할과, 만약 동일한 쿼리가 공유 풀에 존재할 때 얻을 수 있는 성능적 이점을 하드 파싱(Hard Parsing)의 생략 관점에서 서술하시오.

공유풀이 수행하는 역할 : 데이터베이스 메모리 내에서 이전에 실행된 SQL문과 최적의 경로 정보인 실행 계획을 임시로 보관하는 보관함 역할

성능적 이점 : 동일한 쿼리가 이미 공유 풀에 저장되어 있다면, 옵티마이저가 문법 검사, 오브젝트 권한 확인, 실행 계획을 처음부터 다시 새로 생성할 필요가 없다. 구문 분석 속도가 많이 단축된다.


프로시저 내부의 SELECT 문이 실행될 때, 서버 프로세스가 디스크가 아닌 메모리 영역인 데이터베이스 버퍼 캐시(Database Buffer Cache)에서 데이터를 찾았을 때와 찾지 못했을 때의 물리적 처리 과정의 차이점을 데이터 파일(Data File)과 속도 관점에서 비교 서술하시오.


버퍼캐시에 데이터가 있을 때 : 버퍼 캐시에서 데이터를 즉시 읽어 클라이언트로 반환하므로 물리적 I/O자원을 쓰지 않아서 처리 속도가 매우 빠르다.

버퍼캐시에 데이터가 없을 때 : 디스크에 실제 저장된 물리파일인 데이터 파일에서 관련된 테이블 블록을 직접 읽어와야 하므로 속도가 느리다. 이 때 읽어온 데이터는 향 후 재사용을 위해 다시 버퍼 캐시에 임시저장된다.
