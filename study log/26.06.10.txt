- 접근성 및 반응성 점검

- 화면 설계(필요한 화면 설계)

2. UI 화면 설계의 기본 구성요소
ISO 9241-161
- 사용자와 컴퓨터의 상호작용(HCI, Human Computer Interaction)에 대한 국제 표준
- 화면 설계시 자주 사용되는 기본 UI 구성 요소를 정의하고 있다.
- 그 중 기본 구성 요소를 버튼, 체크박스, 입력창 등 51개 항목으로 제시한다.

주요 UI 구성 요소
- 컨테이너 (다른 요소들을 포함하는 요소다)
- 버튼 (명령 실행, 제출, 다른 페이지로 이동)
- 체크박스 (복수 선택 가능한 선택 목록)
- 라디오버튼 (그룹 중에서 하나만 선택)
- 입력창 (사용자 텍스트 입력, 검색 조건 입력)
- 라벨 (제어 요소의 식별자 및 설명을 제공)
- 목록 (항목 정렬)
- 대화상자 ( 사용자 피드백, 확인 요청, 알림 등)
- 폼 (입력, 선택, 클릭과 같은 전체를 구성하는 복합 요소)

3. 유용성 개념 및 적용 원리 파악
3-1. 유용성
- 사용자가 시스템을 활용하여 자신이 원하는 목표를 얼마나 효과적이고 효율적으로 달성할 수 있는지를 의미한다

유용성이 높은 시스템
로그인 버튼을 쉽게 찾을 수 있다.
원하는 메뉴를 바로 찾을 수 있다.
검색 결과가 빠르게 나온다.

유용성이 낮은 시스템
메뉴 위치를 찾기 어렵다.
버튼 이름이 이해되지 않는다.
기능을 사용하기 위해 너무 많은 단계를 거쳐야 한다.

유용한 시스템을 설계할 때 고려사항
- 시스템을 설계할 때 사용자가 생각하는 방식과 실제 시스템이 동작하는 방식의 차이를 줄여야 한다.

회원가입 -> 약관 설명 화면 -> 광고 화면 -> 회원가입 화면 방식으로 작동하면 사용자는 불편함을 느낀다.

사용자가 예상하는 동작과 실제 동작이 다르면 혼란이 발생하고, 오류가 증가하고, 사용시간이 증가하고, 서비스 만족도가 감소할 수 있다.

사용자 모형과 개발자 모형간의 차이 발생
사용자 모형(User Model)
- 사용자가 머릿속으로 생각하는 시스템의 모습
개발자 모향(Developer Model)
- 개발자가 시스템을 구현한 방식

시스템에서 제공하는 실행 가능한 기능과 사용자가 본래 기대한 목적이 서로 다를 때 발생

3-2. 실행 차를 최소화하기 위한 UI 설계 원리
※실행 차 : 사용자가 원하는 행동과 실제 시스템 조작 사이의 차이

사용 의도 파악
- 사용자가 무엇을 이루려고 하는지 명확하게 이해
- 불필요한 부가 기능으로 인해 시스템 성능이 저하되지 않도록 설계
- 사용자가 관심을 가질만한 항목은 눈에 잘 띄는 곳에 두기
- 필요한 순간에 기능이 제공되도록 배치

행위 순서 규정
- 사용자의 행동 과정을 세분화하여 순서대로 명확하게 안내하되, 필요할 경우 사용자가 의도에 맞게 순서를 자유롭게 바꿀 수 있다.
- 한 가지 작업을 완료하는 단계는 가능한 최소화 하기
- 동일한 결과를 다양한 방식으로 얻을 수 있도록 설계에 반영하기
- 행위 순서는 사용자의 기존 경험과 자연스럽게 이어지도록 친숙하게 구성하기

행위의 순서대로 실행
- 사용자가 계획한 작업 절차를 UI 상에서 직관적으로 파악할 수 있도록 제공하여 의도한 행위를 실제로 실행하는 과정에서 혼란이나 어려움이 생기지 않도록 해야한다.
- 불필요하게 많은 상호작용으로 인해 작업 진행이 방해받지 않도록 주의해야 한다.
- 사용자가 의도한 행위에 가장 적합한 입력 장치를 선택하고, 이에 대한 적절한 피드백과 취소 기능을 제공한다.
- 기본값을 알맞게 설정해 불필요한 조작을 줄여야 한다.

3-3. 평가 차를 줄이기 위한 UI 설계 원칙
※평가 차 : 사용자가 시스템의 상태를 이해하는 과정에서 발생하는 차이

키 입력 결과를 사용자가 신속하게 인식할 수 있도록 설계
- 사용자가 어떤 조작을 했음에도 화면에 아무런 변화가 없다면, 사용자가 제대로 입력했는지 의심하게 되고 이는 평가 차리를 키우게 된다.
- 따라서, 가능한 빠르게 시스템이 반응하도록 설계해야 하며, 즉각적인 반응이 어렵더라도 지연을 최소화 해야 한다.
- 또한 사용자가 행위를 했다는 사실이 명확히 들어나도록 직관적인 피드백 제공하기

UI 조작으로 인해 시스템의 상태 변화를 쉽게 파악할 수 있도록 설계
- 사용자의 입력에 따라 시스템이 어떻게 달라졌는지 명확하게 인식할 수 있어야 한다.
- 이를 위해 상태 정보를 단순하고, 이해하기 쉽게 표현해야 하며, 복잡하지 않도록 설계하는 것이 좋다.

사용자의 의도와 시스템 반응 결과의 일치 여부를 쉽게 확인할 수 있도록 설계
- 사용자가 기대한 결과가 실제 시스템을 통해 충족되었는지 쉽게 알 수 있어야 한다.
- 이를 위해 미리보기와 같은 기능을 제공하면 사용자가 예상 결과를 사전에 확인할 수 있어 대부분의 경우 만족도를 높일 수 있다.

4. 스토리보드 작성 방법
- 스토리보드는 서비스 구축 과정에서 디자이너와 개발자가 공통으로 참고하는 최종 산출물이다.
- 정책, 프로세스, 콘텐츠 구성, 와이어프레임, 기능 정의, 데이터베이스 연동 등 서비스 구현에 필요한 핵심 정보가 들어가 있다.
- 화면 설계와 기능 정의를 문서화 하여 개발 과정의 기준이 된다.

4-1. 메뉴 구성도 작성
- 전체 서비스의 메뉴 구조를 정의
- 사용자가 어떤 경로로 서비스를 이용하는지 파악
- 화면 간 이동 관계를 명확하게 정리한다.

4-2. 스타일 확정
- 서비스 전체의 디자인 일관성 유지하기
- 화면마다 다른 디자인이 적용되지 않도록 기준 설정하기
색상, 글꼴,버튼 스타일, 입력창 스타일, 아이콘 및 그래픽

4-3. 설계
- 실제 화면에 표현될 시각적 디자인 콘셉트를 구체화 한다.
