비동기 통신
- 기능을 수행할 동안 기다리지 않고 다른 작업을 수행하는 것

Promise : 나중에 결과를 전달하겠다는 약속 객체

function getMessage(){
	return new Promise();
}

Promise의 3가지 상태
1. Pending : 기다리는 중
2. Fulfilled : 데이터 도착
3. Rejected : 실패

new Promise((resolve, reject) => { 
	resolve(비동기로 작업할 내용을 작성);
	reject(비동기로 작업할 내용을 작성);
});

resolve : 성공 결과를 전달하는 함수 -> promise의 상태를 fulfilled로 바꾼다.
reject : 실패 결과를 전달하는 함수 -> promise의 상태를 rejected로 바꾼다.

promise의 상태가 fulfilled로 바뀌게 되면 then() 함수가 실행이 된다.
promise의 상태가 rejected로 바뀌게 되면 catch() 함수가 실행이 된다.

then(response) -> 결과를 받게된다.

fetch api
- javascript에서 서버에 요청을 보내는 대표적인 비동기 함수이다.

fetch("주소")
	.then(response => response.json())
	.then(data => 데이터 활용하는 코드);

response는 실제 데이터만 들어있지는 않다.
상태코드 2xx, 3xx, 4xx, 5xx
응답 성공 여부
헤더 정보
본문데이터 읽기 기능

실제 데이터는 response.json()을 해야 얻을 수 있다.

fetch()는 서버가 404나 500을 보내도 무조건 catch()로 가지는 않는다.
그래서 response.ok를 활용하는 습관이 필요하다.

테스트 기법 선정
1. 휴리스틱 평가
- 소수의 UI/UX 전문가들이 이미 정의된 '경험적인 대원칙'을 바탕으로 제품을 검토하고 사용성 문제를 찾아내는 이론, 경험 기반의 평가 방법이다.
장점
- 상대적으로 비용이 적게들고, 짧은 시간 내에 시스템의 중요한 문제점을 발견할 수 있다.
- 프로젝트 수행 과정 전반에 걸쳐 언제든 테스트가 가능하므로, 개발 초기에 문제를 빠르게 잡아낼 수 있다.

단점
- 구체적이고 계량적인평가자료를 만들기 어렵다.
- 전문가가 바라보는 시각과 실제 사용자가 바라보는 시각이 다를 수 있다.
- 평가에 참여하는 전문가의 역량과 능력에 따라 결과의 질이 달라진다.

2. 페이퍼 프로토타입 평가
- 코딩이나 본격적인 디자인을 하기 전, 종이에 화면구조를 그리고 사람이 직접 컴퓨터 역할을 하며 종이를 바꿔 끼워주는 방식으로 인터렉션을 시뮬레이션 하는 기법

장점
- 가장 빠르게 프로토타입을 만들 수 있어 제품의 전반적인 콘셉트와 흐름을 신속하게 검증할 수 있다.
- 완성된 결과물이 아니기 때문에, 평가에 참여하는 사람들이 최종 제품에 대한 부담이나 기대를 갖지 않고 더욱 자유롭게 의견과 개선점을 제안할 수 있다.

단점
- 현실감 있는 테스트를 위해 UI 스케치를 넘어 정교한 프로토타입과 테스트 설계까지 초함하게 될 경우, 한 달 이상의 긴 시간이 소요될 수 있다.

3. 선호도 평가
- 제품,서비스에 대한 사용자의 주관적인 선호도와 감성을 과학적, 객관적으로 측정하고 예측하는 평가 방법

장점
- 사용자의 감성과 니즈를 객관적으로 해석하여 디자인 의사결정에 반영할 수 있다.
- 점수(rating), 순위(ranking), 태도/속성 기반 선호도 등 다양한 측정 방법을 적용해 데이터를 수집할 수 있다.
단점
- 사용자의 주관적인 취향이나 감성에 의존하므로, 실제 태스크 수행 능력이나 시스템의 기능적 오류를 잡아내기에는 한계가 있다.

4. 성능 평가
- 개발 마지막 단계에서 사용자가 실제로 제품이나 서비스를 이용해 특정 태스크를 수행하게 한 뒤, 학습성, 효율성, 기억 용이성, 오류, 만족도 등을 정량적으로 평가하는 방법

장점
- 실제 사용자의 관점에서 회원가입 완료 시간(효율성), 오클릭 횟수(오류)등 계량적이고 구체적인 데이터를 얻을 수 있다.
- 실제 배포 전, 개발자가 간과하기 쉬운 시스템의 치명적인 문제나 전문 용어 남용 등의 실수를 확실하게 검증할 수 있다.

단점
- 대체로 개발 최종 단계에서 진행되므로, 이때 발견된 구조적인 문제를 수정하려면 많은 비용과 시간이 수반될 수 있다.
