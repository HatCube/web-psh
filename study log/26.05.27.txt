논리 데이터모델 검증하기
- 필요한 데이터가 빠지지 않았는지(입력, 출력, 연계 데이터)
- 현재 시스템의 데이터구조를 파악했는지(기본 DB구조, 백업, 공통코드, 외부 연계 데이터)
- 사용자 요구사항이 반영되었는가
- 데이터 흐름이 명확한가
- 데이터 모델이 적절한가
- 접근권한과 통제가 명확한가

물리 데이터저장소 설계하기
- 논리 데이터저장소 설계를 바탕으로 목표 시스템의 데이터 특성을 반영하여 최적화된 물리 데이터저장소 설계

반정규화
- 정규화를 조금 포기하고, SELECT 성능을 높이기 위해 데이터의 중복을 감수하고, 테이블 구조를 바꾸는 설계 방법

반정규화를 하는 이유
- 테이블이 너무 잘게 쪼개져있어서 JOIN을 너무 많이해야해서 
- 데이터가 적을때는 괜찮은데 데이터가 많아지면 SELECT 속도가 느려진다.
- 자주 조회하는 데이터를 일부러 합치거나 중복해서 저장한다.

핵심개념
- JOIN을 줄여 빠르게 조회속도 향상
- 합계, 평균을 미리 저장하여 계산 비용 감소
- 자주쓰는 속성은 테이블로 따로 만들어 분리
- 항상 같이 쓰는 테이블을 하나로 합친다.

### 문제3. 다음 논리 모델 관계를 분석하여 **외래키를 추가할 테이블, 외래키명, 필요한 제약조건**을 작성하시오.

**제공 관계**

```
고객 1 : 예약 N
→ 예약은 반드시 고객에 연결되어야 한다.

객실 1 : 예약 N
→ 예약은 반드시 객실에 연결되어야 한다.

예약 1 : 결제 1
→ 결제는 반드시 예약에 연결되어야 한다.

예약 1 : 체크인 1
→ 체크인은 반드시 예약에 연결되어야 한다.
```

**외래키 칼럼명 규칙**

- 참조하는 테이블의 기본키 이름을 그대로 사용한다.
- 기본키는 `_id` 형식을 사용한다.

**제약조건 규칙**

- 관계 설명에 **"반드시"** 라는 조건이 있으면 `NOT NULL`을 추가한다.
- `1:1 관계`는 외래키에 `UNIQUE`를 추가한다.
- 관계 연결은 `FOREIGN KEY`를 사용한다.

```
② 객실 1박 요금 저장
③ 실제 체크인 시간 저장
④ 예약 상태 저장

```

데이터가 수정되면 인덱스도 같이 수정을 해야 한다.

지나치게 많은 인덱스는 오버헤드가 된다.
오버헤드 : 추가 부담
>인덱스가 많으면 조회는 빨라질 수 있지만,변경작업은 느리다.

예를 들어 회원 1명을 추가할 때 테이블에만 저장하는것이 아니라 여러 인덱스도 함께 갱신해야 한다.
ex)인덱스가 5개라면, 회원 데이터 1건을 넣을때 인덱스 5개도 함께 관리해야한다.

인덱스는 추가 저장공간이 필요하다.
>테이블과 별도로 저장되는 검색 구조이다.
>인덱스를 만들면 저장공간을 추가로 사용한다.
>그래서 무조건 많이 만드는것은 좋지않다.

넓은 범위를 인덱스로 처리하면 오히려 느려질 수 있다.
인덱스는 적은 데이터를 빠르게 찾을때 효과적이다.
>하지만 전체 데이터의 많은 부분을 조회하면 오히려 전체 테이블을 읽는것이 더 빠를 수 있다.

select * from orders
WHERE order_date BETWEEN '2020-01-01'AND'2026-12-31'

이 조건이 주문의 90%를 조회한다면 인덱스 효과가 낮아진다.
이런경우 DB는 인덱스보다 테이블 스캔을 선택할 수 있다.

인덱스와 테이블 데이터의 저장공간을 적절히 분리한다.
>대규모 DB에서는 테이블 데이터와 인덱스 데이터를 물리적으로 분리해서 저장하기도 한다.
>이렇게하면 디스크 입출력 성능이나 관리측면에서 유리할 수 있다.

뷰(view)
>실제 데이터를 저장하는 테이블이 아니라 select 문을 저장해둔 가상의 테이블
>자주 사용하는 조회 SQL에 이름을 붙여 둔 것

뷰가 필요한 이유
>복잡한 SQL을 단순하게 사용할 수 있다.
ㄴ여러 테이블을 join하는 SQL은 길고 복잡하다.

특정 컬럼만 보여줄 수 있다.
ㄴ개인정보,급여와 같은 민감한 정보는 제외하여 뷰를 만들면 필요한 컬럼만 공개할 수 있다.

>조건에 맞는 데이터만 보여줄 수 있다.
ㄴ활성 회원만 보여주는 뷰

뷰 속성
replace
>뷰가 이미 존재하는 경우 기존 뷰를 새 내용으로 다시 생성하는 속성
create view active_member_view as 
select member_id,name,email
from member
where status = 'active';

force
>기본 테이블이 존재하지 않아도 뷰를 생성하는 속성
create force view member_view as 
select member_id,name from member;

이 경우 oracle에서는 테이블이 없어도 view 정의 자체는 만들수 있다.
다만 실제로 조회하려고 하면 오류가 발생할 수 있다.

왜냐하면 실제 테이블은 없기 때문이다.

noforce
기본 테이블이 존재할 때만 뷰를 생성하는 속성
create noforce view member_view as 
select member_id,name,from member;
>oracle에서 기본값은 보통 noforce이다.
>일반적으로 참조할 테이블이 있어야 뷰를 만들 수 있다.

with check option
>뷰의 조건을 만족하는 행만 insert 또는 update를 할 수 있도록 제한하는 속성 

>활성 회원만 저장하는 view
create view active_member_view as
select member_id,name,status
from member
where status = 'active'
wirh check option

가능한 수정 
update active_member_view
set name = '김영희'
where member_id = 1;

상태를 건드리는 수정은 할 수 없다.
update active_member_view
set status = 'inactive'
where member_id = 1;

> 이 뷰의 조건을 벗어나는 수정은 허용하지 않는다.

with read only
>뷰를 조회 전용으로 만드는 속성
create view active_member_view as 
select member_id,name,status
from member
where status = 'active'
with read only;

뷰 설계시 고려사항 
뷰는 결국 테이블을 조회한다.
>실제 데이터를 저장하지 않는다.
>실제로는 내부에서 원본 테이블의 select 가 실행되는 것과 같다.
>그래서 뷰를 사용한다고 해서 무조건 빨라지는 것은 아니다.
오히려 뷰 안의 SQL이 복잡하면 성능 문제가 생길 수 있다.

복잡한 뷰는 성능이 느려질 수 있다
>member,orders테이블을 조인하는 쿼리문을 view로 만든다.
>뷰를 조회하게 되면 실제로는 member와 orders 테이블을 조인하고 집계한다.
>데이터가 많으면 이 작업은 느려질 수 있다.

뷰 안의 select 조건도 인덱스를 고려해야한다.
>뷰 안에 조건이 있다면,해당 조건 컬럼에 인덱스가 있는지 확인해야한다.
create view active_member_view as
select member_id,name,email
from member
where status = 'active'
>이 뷰는 status 조건을 사용한다.
>따라서 status 컬럼이 자주 조회되고 분포도가 적절하다면 인덱스를 고려할 수 있다.
다만 status 같이 active와 inactive처럼 종류가 적고 대부분이 active라면 효과가 낮을 수 있다.

뷰를 사용하는 SQL의 where절도 중요하다.
>뷰 안의 조건이 부족하더라도,뷰를 조회하는 SQL에서 좋은 조건을 줄 수 있다.
create view order_view as
select order_id,member_id,order_date,total_price
from orders;
>모든 주문을 보여주는 뷰
>그런데 실제 조회할때는 다음처럼 조건을 준다.
select *
from order_view
where member_id = 10;
>이 경우는 실제로는 orders.member_id를 조건으로 조회하게 된다.
>따라서orders.member_id에 인덱스가 있으면 성능이 좋아질 수 있다.

파티션
>큰 테이블을 여러개의 작은 영역으로 나누어 저장하는 방법

파티션 설계
>큰 테이블을 여러개의 작은 영역으로 나누어 저장하는 방법
>주문 데이터가 1억 건 있다고 가정
orders 테이블
주문번호
회원번호
주문일자
상품명
가격

만약 모든 데이터를 하나의 테이블에 저장하면 검색할 때 DB가 엄청난 데이터를 확인해야한다.
시간이 오래걸리기 때문에 데이터를 나눈다.
orders_2024
orders_2025
orders_2026

파티션이 필요한 이유
>대용량 시스템에서는 데이터가 계속 증가한다.
>데이터가 많아지면
조회속도 저하
백업시간 증가
복구시간 증가
디스크 사용량 증가

파티션 종류
1.범위 분할 
>지정한 컬럼 값의 범위를 기준으로 데이터를 나누는 방법
주문테이블의 데이터를 주문일자 기준으로 나눈다.
주문 테이블
주문번호 	주문일
101		2024-01-10
102		2025-03-01
103		2026-06-12

반정규화 분할 > 테이블을 진짜로 여러개로 만듬
파티션 > 하나의 테이블 내부에서 저장 영역을 나눈다.

create table orders(
	order_id int,
	order_date date
)
partition by range(year(order_date))
(
partition p2024 values less then(2025),
partition p2025 values less then(2026),
partition p2026 values less then(2027)

);
>파티션을 만들어도 사용자는 일반 테이블 처럼 사용한다.
>원래 하던것처럼 CRUD 작업을 한다.
