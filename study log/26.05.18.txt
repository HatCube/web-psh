요구사항 확인
>요구사항이 올바른지 점검하는 과정이다.

확인과 검증
Validation(확인)
>만드려는 시스템이 고객이 실제 원하는 것인지 확인하는 것
>요구사항 문서가 아무리 깔끔하게 작성되어 있어도 고객이 원하는 기능이 아니면 의미가 없다.
ex) 학생 출결을 관리하고 싶다.

출결 등록/조회/수정 기능을 포함했다. > ok
모바일을 원했는데 pc화면만 설계됨 > 실패
관리자 승인 기능을 원했는데 자동 승인으로 설계됨 > 실패

Verification(검증)
>작성된 요구사항 문서가 올바르게 작성되었는지 검사하는 것
>고객이 원하는 방향이 맞는지도 중요하지만,그 요구사항이 문서로 정확하고 일관성 있게 정리되어 있어야한다.

요구사항 문서 검증 시 확인 항목
1.이해 가능성 : 읽는 사람이 같은 의미로 이해할 수 있는가
2. 일관성 : 내용끼리 서로 충돌하지 않는가
3.완정성 : 필요한 내용이 빠지지 않았는가
4.표준 준수 : 회사 규칙을 따르는가


클러스터
>여러대의 컴퓨터(서버)를 하나처럼 묶어서 사용하는 기술

오픈소스
>소스코드가 공개되어 있고,자유롭게 사용 가능한 프로그램
>아무 제한 없이 사용이 되는것은 아니고 라이선스 규칙을 따라야 한다.

장점
비용절감 : 무료로 사용가능
자유로운 수정 : 기능 추가 가능 
커뮤니티 활성화 : 자료가 풍부하다
빠른 업데이트 : 개발자의 참여가 많다.

단점
공식 지원 부족 : 제조사 지원 약할 수 있음
설정 난이도 : 직접 설정 필요
라이선스 확인 필요 : GPL등 확인 필요

로그인 기능 개발
개발자 A > 로그인 기능 
개발자 B > 회원 로그인 처리 기능
개발자 C > user login
개발자 D > 로그인 모듈

위와같이 작성했을때 문제점
>같은기능인지 구분 어렵다
>문서를 찾기 어렵다
>유지보수가 어렵다
>협업 효율이 감소한다

표준 준수에서 확인하는 것들
1.문서 형식 표준
회사에서 요구사항 문서를 다음과 같이 정했다고 가정해보자
요구사항 id, 기능명 ,설명, 우선순위,담당자

REQ - 001
기능명 : 로그인 
설명 : 사용자는 로그인 할 수 있다
우선순위 높음 
담당자 : 홍길동

2.요구사항 번호 규칙 
회사 규칙 : 
REQ - 001
REQ - 002
REQ - 003
...

REQ - 001 : 로그인
REQ - 002 : 회원가입
REQ - 003 : 게시글 등록

3. 용어 표준 
문서 일부 : 회원 
다른 문서 : 사용자 
다른 문서 : 고객
>같은사람인지 다른사람인지 모른다.

전체 문서 : 회원(member)로 통일하여 사용한다.

요구사항 관리 툴
요구사항 문서를 엑셀이나 한글 파일로만 관리하지 않고, 변경 이력, 승인 상태,담당자, 추적관계까지 체계적으로 관리하는 도구

요구사항 관리 툴을 사용하는 이유
>요구사항은 프로젝트 중간에 계속 바뀐다.
누가,언제, 왜 바꿨는가
승인 되었는가
어떤 기능에 영향이 있는가
테스트도 바뀌어야 하는가
위와같은 내용을 관리해야 한다.

사람이 파일로 관리하면 빠트리기 쉽기 때문에 요구사항 관리 툴을 사용한다.

요구사항 관리 툴이 하는 일
요구사항 등록 : 요구사항을 항목 단위로 저장
버전 관리 : 변경 전 / 후 이력 관리
변경 이력 관리 : 누가, 언제, 무엇을 바꿨는지 기록
승인 관리 : 검토 중, 승인, 반려 상태 관리
추적성 관리 : 요구사항과 설계 , 개발, 테스트 연결
담당자 관리 : 누가 처리할지 지정
우선순위 관리 : 중요도 설정
산출물 관리 : 요구사항 정의서, 명세서 테스트 문서 관리

형상 관리와 요구사항 관리의 관계

요구사항 확인 기법
작성된 요구사항이 올바른지,빠진것은 없는지,고객이 원하는 내용이 맞는지 확인하는 방법

1.요구사항 검토
>여러 사람이 모여 요구사항 문서를 읽고 문제를 찾는다.
오타,잘못된 가정,애매한 표현,누락,모순,표준 위반점을 찾는다.

>검토자 그룹을 어떻게 구성하느냐가 중요하다
예를 들어 고객 중심 프로젝트에서는 검토자 그룹에 고객 대표자가 1명 이상 포함되어야 한다.

>검토는 시스템 정의서,시스템 사양서,소프트웨어 요구사항 명세서를 완성한 시점에서 이루어진다.

시스템 정의서
>무엇을 만드는지 큰 방향을 정의하는 문서

시스템 사양서
>정의서보다 조금 더 구체적인 문서

소프트웨어 요구사항 명세서
> 개발자가 실제 구현할 수 있을 정도로 상세하게 적은 문서

2.프로토 타이핑(prototyping)
>최종 제품을 만들기 전에 일부 기능이나 화면을 미리 만들어 사용자가 확인하도록 하는 방법
>소프트웨어 요구사항에 대하여 소프트웨어 엔지니어가 해석한 것을 확인할 수 있다.
>처음에는 생각하지 못했던 새로운 요구사항을 도출할 수 있다.
