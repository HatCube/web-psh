서버 프로그램 구현
어플리케이션 설계를 기반으로 개발에 필요한 환경을 구축하고, 프로그래밍 언어와 도구를 활용하여 공통 모듈, 업무 프로그램과 배치 프로그램을 구현할 수 있다.

공통 모듈
- 어플리케이션 전체에서 반복적으로 사용되는 기능을 독립적인 프로그램 단위로 분리한 것
ex) 로그인 및 권한 체크, 로그의 기록, 날짜 계산, 문자열 처리...

업무 프로그램
- 사용자가 화면을 통해 요청한 비즈니스 로직을 실시간으로 처리하는 프로그램

배치 프로그램
- 사용자와 상호작용 없이, 대량의 데이터를 정해진 시간에 일괄적으로 처리하는 프로그램

개발환경 준비
응용 소프트웨어 개발에 필요한 하드웨어 및 소프트웨어의 필요 사항을 검토하고, 이에 따라 개발 환경에 필요한 준비를 수행할 수 있다.

응용소프트웨어
- 특정 사용자 요구를 충족시키기 위해 컴퓨터 시스템에서 실행되는 프로그램
- 운영제와 같은 시스템 소프트웨어와 달리, 사용자가 필요로 하는 구체적인 업무 처리나 기능 수행을 목적으로 한다.

응용 소프트웨어의 특성
1. 사용자 목적 중심
- 사용자 혹은 조직의 문제 해결, 업무 자동화, 정보 처리 등을 지원한다.
2. 도메인 특화
- 특정 산업, 업무, 조직의 업무 규칙을 반영한다
3. 프로그램 구성 요소의 집합
- 단일 어플리케이션이라도 다수의 모듈(공통 기능 모듈, 업무 처리 모듈, 인터페이스 연계 모듈 등)로 구성된다.
4. 운영환경과 밀접한 연관
- 운영체제, 데이터베이스, 네트워크, 보안 환경과도 유기적으로 통합되어 작동한다.

응용 소프트웨어의 유형
데스크탑 응용 SW
- PC기반 독립 실행 프로그램
웹 기반 응용 SW
- 웹 브라우저에서 실행되는 서비스
모바일 응용 SW
- 스마트폰, 태블릿에서 실행되는 앱
임베디드 응용 SW
- 특정 하드웨어 장비에 내장되어 구동
클라우드 응용 SW
- SaaS 형태로 제공되는 응용 SW
(Software as a Service)
대규모 엔터프라이즈 SW
- 기업 전사 업무 처리용 대형 시스템 (ERP, CRM, SCM)
AI/데이터 기반 SW
- 인공지능, 빅데이터 분석을 수행

개발환경
- 응용 SW 개발 환경은 응용 소프트웨어를 개발,빌드,테스트,배포하기 위해 필요한 하드웨어, 소프트웨어, 네트워크, 개발 도구 및 집합을 의미하며, 개발자가 소프트웨를 효율적이고 안정적으로, 높은 품질 수준을 유지하여 개발할 수 있도록 지원하는 인프라와 도구들의 통합된 시스템으로 구성된다.
- 개발 프로세스 전체(코딩, 컴파일, 디버깅, 형상관리, 테스트, 배포 등)를 지원하며, 개발 환경은 운영환경과 유사하게 맞추어 구성하는 것이 원칙이다.

개발환경의 유형
로컬 개발 환경
- 개발자의 PC에서 직접 구축,운영
서버 기반 개발 환경
- 중장 개발 서버에 개발 환경 구축, 다수 개발자 공유
가상화 개발 환경
- 가상 머신 또는 컨테이너로 독립적 환경 구축(DOCKER)
클라우드 개발 환경
- 클라우드 서버로 개발 환경을 임대/운영(AWS)
DevOps 통합환경
- CI/CD, 자동 배포 등을 포함한 개발-운영 통합 환경(Jenkins)

개발환경 준비하기
목표 시스템의 운영환경을 분석한다.
1. 목표 시스템의 환경 파악하기
- 제안 요청서, 제안서, 사업 수행 계획서, 요구사항 정의서, 시스템 아키텍처, 어플리케이션 아키텍처 등 분석 및 설계 시의 산출물을 분석하여 목표 시스템의 환경을 분석한다.

2. 목표시스템의 운영환경을 분석한다
- 목표 시스템의 운영환경을 하드웨어, 운영체제, 데이터베이스/파일, 네트워크, 미들웨어, 보안시스템, 모니터링/관리 도구, 배포, 운영 자동화 도구 등의 구성 요소별로 구분하여 파악한다.

3. 목표시스템의 개발 환경을 분석한다.
- 목표 시스템의 개발 환경을 운영환경을 고려하여 유사하게 구성 요소별로 분석한다.

4. 목표시스템의 응용 SW의 요구사항을 분석한다.
- 목표시스템의 요구사항을 분석/설계, 기능, 성능, 인터페이스, 보안, 테스트, 품질 등의 분야별로 구분하여 분석한다.

개발 언어 및 하드웨어 사양을 고려한 구현 도구를 선택한다.
1. 개발 언어의 선정
1-1. 선정 기준
- 적정성 : 대상 업무의 성격, 개발하고자 하는 시스템이나 응용 프로그램의 목적에 적합해야 한다.
- 효율성 : 프로그램의 개발 및 실행 효율성을 고려해야 한다.
- 이식성 : 다양한 OS, 플랫폼에서 동작 가능해야 하며, 클라우드/컨테이너 환경과 호환되어야 한다.
- 친밀성 : 개발자가 해당 언어를 이해하고 쉽게 사용할 수 있어야한다.
- 범용성 : 과거 개발 실적과 커뮤니티 지원, 유지 보수 가능성, 표준화 여부 등이 충분해야 한다.
- 생태계 : 다양한 라이브러리, 프레임워크, 툴 지원이 풍부하고, 개발 커뮤니티가 활성화 되어 있어야 한다.
- DevOps/클라우드 적합성 : 자동화, 배포, 클라우드 서비스와 연계가 용이해야 한다.

1-2. 추가 고려 요소
- 개발 인력의 경험, 숙련도
- 운영환경(운영체제, 네트워크 환경)
- 유지보수 비용, 라이선스 비용
- 프로젝트의 규모와 기간
- 클라우드 네이티브 적합성

1-3. 개발 언어의 최신 트렌드
- Server Side
Java : 여전히 엔터프라이즈 분야의 표준, Springboot와 함께 사용
Python : AI/ML, 데이터 분석뿐 아니라 백엔드 개발에서도 널리 사용
JavaScript/Node.js : 경량화 서비스, API 서버 구현에 적합
Go : 클라우드 네이티브, 마이크로서비스 환경에서 각광
C# : Windows뿐 아니라 Linux에서도 운용이 가능하다
Kotiln : Java 대체 언어로 사용 증가
Rust : 성능과 메모리 안정성을 요구하는 시스템 개발에 부각

- Client Side
JavaScipt/TypeScript : React, Vue, Angular 등 현대 웹 프레임워크 기반 개발의 표준
Flutter/Web : 크로스 플랫폼 웹앱 개발

- Moblie
Swift : iOS 앱 개발 표준
Kotlin : Android 앱 개발 표준
Flutter, React Native : 크로스 플랫폼 앱 개발에서 인기

2. 개발 도구의 선정
2-1. 통합 개발 환경의 종류와 특징
IntelliJ IDEA : JetBrains에서 개발, Java및 코틀린 개발의 표준과도 같은 강력한 IDE, 유료버전과 무료버전이 존재
Visual Studio Code : Microsoft에서 개발, 거의 모든 프로그래밍 언어를 지원한다.
Ecilpse : 이클립스 재단에서 개발, Java 개발에 주로 쓰인다.
PyCham : JetBrains, 파이썬 개발에 특화된 프로그램, 유료랑 무료가 존재
Rider : JetBrains, C# 및 .NET 기반 개발을 위한 유료 IDE
Xcode : Apple, 맥 전용툴, iOS, Mac 앱 개발
Android Studio :  Google, 안드로이드 앱 개발
Github Codespaces : Github에서 개발, 다수의 언어를 지원하나 쓴만큼 과금이 됨
Gitpod : Gitpod에서 개발, 자동으로 개발 환경을 준비해주는 클라우드 기반의 자동화 IDE

2-2. 최신 개발 환경 트렌드
클라우드 IDE
- 원격 개발, 협업 개발 가속화가 가능하며, 인프라 준비 없이 즉시 개발 환경 구축이 가능하다
DevOps 연계
- IDE에서 직접 빌드, 배포, 테스트 연동이 가능하며, CI/CD 파이프라인 관리 툴과 연동이 가능하다
AI 개발 지원
- GitHub Copilot, Codeium, Tabnine 등 AI 코딩 어시스턴트 활용이 가능하다.
경량화 IDE
- VS CODE가 사실상 표준 개발 툴로 자리를 잡고, WEB기반 IDE 확대가 가능하다.

2-3. 최신 개발 환경 도구 선정 시 고려 사항
- 운영환경과의 호환성
- 플러그인 생태계
- 클라우드/DevOps 통합 기능
- 협업 기능(코드 리뷰, 공동 편집)
- 라이선스 비용 및 유지보수
- 성능 및 리소스 요구

2-4. 통합 개발 환경의 선정
운영환경과 개발 환경간의 언어,도구 일치가 매우 중요하며, 배포 및 유지보수 리스크를 최소화할 수 있도록 최신 기술 동향을 반영해 개발 환경을 선정한다.
Server Side -> Java, Node.js, Go
Client Side -> TypeScript+React, Vue.js
Mobile -> kotlin, Swift, Flutter
IDE -> Intellij, VS code, PyCharm, Xcode, Android Studio
Cloud IDE -> GitHub CodeSpaces, Gitpod

3. 프로그램의 배포 및 라이브러리 관리를 위한 빌드 도구를 선정
3-1. 빌드 도구 Gradle, Maven, Ant의 특징
Gradle
- 빠른 빌드 속도와 캐싱 지원
- Groovy/Kotlin 스크립트 사용 가능
- 대규모 프로젝트, 멀티 모듈 지원에 유리
- Anroid 공식 빌드 도구
Maven
- 프로젝트 전체 정보를 표준화된 POM파일로 관리
- 라이프사이클 기반 빌드 수행
- 의존성 관리가 체계적이다
- 대규모 엔터프라이즈 프로젝트에 여전히 많이 사용
Ant
- 프로젝트별 개별 Target 기반 실행
- XML 기반 빌드 스크립트 작성
- 유연성은 높으나 표준화, 재사용성이 낮다
- 최신 프로젝트보다는 유지보수성이 낮다

npm
- Node.js 환경에서 표준 빌드, 배포관리
- 프론트엔드에서 빌드 스크립트 관리
- 스크립트 실행, 의존성 관리, 패키지 배포 지원

Webpack/Vite/Rollup
- 프론트엔드 앱 번들링 도구
- 코드 스플리팅, 라이브러리 트리 셰이킹
- 최신 JS 빌드 성능 최적화

3-2. 최신 빌드 도구 트렌드를 파악한다
DevOps & CI/CD 통합
- 단순 빌드 도구를 넘어 자동화된 파이프라인 통합 추세
- 빌드, 테스트, 배포까지 자동화
- Jenkins, GitHub Actions, GitLab CI/CD, CircleCI
멀티 플랫폼 빌드
- 클라우드 환경, 컨테이너 기반 배포
- Docker, Kubernetes와 연계
프론트엔드 빌드 최적화
- Webpack, Vite, Rollup으로 JS 번들링, 빌드 시간 단축

3-3. 빌드 도구의 선정 기준을 정한다
프로젝트 특성 : 빌드 도구의 친밀도, 팀 경험, 적용 사례를 고려
생산성 : 빌드 속도, 캐시 기능, 분산 빌드 지원 여부
표준화/통합성 : DevOps 파이프라인 통합, 다른 도구와 연계 용이성
플랫폼 호환성 : 멀티 OS, 컨테이너, 클라우드 환경에서 사용가능 여부
의존성 관리 : 외부 라이브러리 관리의 편의성
라이선스/비용 : 오픈소스 여부, 상용 비용 고려

4. 개발 인원을 고려한 형상관리 도구를 선정한다.
4-1. 형상관리 도구의 종류
Git
SVN
CVS

4-2. 최신 형상관리 도구 트렌드
Git 중심의 분산 버전 관리
- 현업 개발에서 Git이 사실상 표준이다.
로컬 개발 -> 중앙 저장소 -> CI/CD 파이프라인으로 연계
- 협업 개발 필수 기능
Branch 전략(git flow ,github flow)
ㄴ 하나의 소스코드에서 여러 개발자가 독립적으로 다른 기능을 개발할 수 있도록 선을 나누는 규칙
Pull/Merge Request
ㄴ내가 만든 코드 변경사항을 프로젝트의 메인 코드에 합쳐달라고 요청하는 절차
Code Review
ㄴ 동료 개발자가 작성한 코드를 메인 소스에 합치기 전에 함께 읽어보며 검토하는 과정
GitOps
ㄴ Git을 시스템 인프라 관리의 단일 진실 공급원으로 삼는 방식
※단일 진실 공급원 : 정보가 여러 군데 흩어져 있으면 서로 내용이 달라서 무엇이 진짜인지 알 수 없게 되므로, 모든 사람이 하나의 데이터만 바라보고 신뢰하자

클라우드 기반 형상관리
- SaaS로 형상관리 도구 활용 증가
- 주요 플랫폼 : GitHub, GitLab
- 코드 저장소 + CI/CD + 이슈 관리 통합, 보안, 권한 관리 강화, AI 코드 리뷰, 자동화 기능 제공

GUI 기반 툴 강화
- CLI만 사용하는 시대는 지나갔다

보안 스캐닝 통합
- Git 저장소 스캔 : Secret 검출(개발자가 코딩을 하다가 외부 api 키, 데이터베이스 비밀번호, AWS 접속 키 같은 중요한 비밀정보를 실수로 코드에 적어두고 Git에 올리는 대참사가 종종 발생한다)
- Dependency Vulnerability Check(의존성 취약점 검사) : 외부 라이브러리들의 목록과 버전을 전 세계 보안 데이터베이스와 대조해서 위험하니 업데이트 해라 알려주는 보안 진단 서비스

클라우드 사용 여부 : 클라우드 SaaS(GitHub)로 협업 용이

교육/숙련도 : 팀의 Git 이해도와 학습곡선 고려

5. 프로젝트 검증에 적합한 테스트 도구를 선정한다.
테스트 계획 및 관리 (테스트의 시작부터 끝까지 계획을 세우고 버그를 관리하는 도구) : Jira, Testrail, Azure Test Plans

테스트 실행 및 자동화(사람이 직접 테스트 하지 않고, 코드가 코드를 자동으로 검사하게 만드는 도구) : Junit

소드코드 분석(프로그램을 실행하기 전후로 나누어 소스 코드 자체의 결함이나 성능을 뜯어보는 도구) :SonarQube, Snyk

파이프라인 및 모니터링(개발환경과 테스트를 연결하고, 시스템이 잘 돌아가는지 지켜보는 도구) : Jenkins, Github Actions

개발환경 구축
1. 개발 하드웨어 환경
- 개발 환경의 하드웨어는 운영환경과 유사한 구조로 구성하는 것이 원칙이며, 안정성, 확장성, 성능을 고려하여 구축해야한다.
- 특히 최근에는 클라우드, 가상화, 컨테이너 기술 등의 확산으로 하드웨어 인프라의 구성이 다양해지고 있다.
1-1. 클라이언트 환경 구성
- 클라이언트 서버 시스템에서 제공하는 서비스를 활용하기 위해 사용자와의 인터페이스를 제공하는 하드웨어로, 일반적으로 PC, 웹 브라우저 환경, 모바일 디바이스가 클라이언트로 활용된다.
클라이언트 환경 최신 구성 동향
- 최신 동향에서는 고사양 개발용 노트북, 다중 모니터 환경, 클라우드 기반 개발 환경 접속용 단말, 태블릿 등이 함게 사용되며, 특히 원격,재택 개발 환경에서나 클라우드 개발 워크스페이스 접속 단말로 클라이언트가 활용된다.
클라이언트 구성 방안
- Windows/MacOS/Linux 기반 개발용 노트북(16~32gb 메모리, ssd)
- 태블릿, iPad Pro, Android Tablet
- 웹 브라우저
- 모바일 디바이스

1-2. 서버 환경 구성
- 서버 환경은 목적에 따라 어플리케이션 서버, 데이터베이스 서버, 파일 서버, 개발 빌드 서버, 테스트 서버 등으로 나눌 수 있다.
- 최근 개발 환경에서는 온프레미스 환경 외에 클라우드 기반 서버가 적극적으로 활용되며, 특히 컨테이너 오케스트레이션 환경과 CI/CD 파이프라인 서버가 필수 요소로 자리잡고 있다.

웹 서버
- 클라이언트에서 요청하는 서비스의 속도를 향상시키기 위해 HTML,CSS,이미지 등의 정적 콘텐츠를 제공하는 웹서버 어플리케이션이 설치되는 하드웨어
- NginX, Apache HTTP Server, Caddy Server

웹 어플리케이션 서버
- 동적 웹서비스를 제공하기 위해 Tomcat, Undertow, IIS등 미들웨어인 WAS와 서비스에 관련된 어플리케이션이 설치되는 하드웨어
- 최근에는 마이크로서비스 아키텍처로의 전환에 따라 경량 WAS, Node.js서버, Springboot 기반 어플리케이션 서버가 증가하고 컨테이너 기반 배포가 표준화되고 있다.

데이터베이스 서버
MySQL, Oracle, Microsoft SQL Server 등 데이터베이스가 설치되는 하드웨어
- 최근에는 클라우드 기반 관리형 데이터베이스 서비스와 NoSQL 계열 데이터베이스도 많이 활용되고 있다.
- 운영환경과 동일한 DB엔진과 버전을 개발환경에도 맞추는것이 중요하다

파일 서버
- 서비스 제공을 위해 파일을 저장하고 공유하기 위한 파일 저장 서버 하드웨어
- 최근에는 온프레미스 파일 서버 외에 클라우드 스토리지(S3)가 개발 환경에서도 많이 사용된다.
- 또한 개발 빌드 산출물 관리에는 아티팩드 저장소가 필수적으로 구성된다.

최신 개발 하드웨어 트렌드
- 로컬 장비보다는 클라우드 기반 개발 환경 사용 증가
- 컨테이너 환경이 표준화되어 개발 서버, WAS, DB까지 컨테이너로 관리
- 클라우드 IDE 이용한 원격 개발 확대
- 관리형 서비스 활용 증가로 로컬 하드웨어 의존도 감소
- 빌드/배포 자동화를 위한 고성능 CI/CD 서버 필수화

2. 개발 소프트웨어 환경
2-1. 운영체제
- 하드웨어 운영을 위한 운영체제로서, Windows, Linux, UNIX등의 환경으로 구성된다.
- 현재는 특히 클라우드 환경에 최적화된 OS 선택이 중요하며, 개발 환경은 운영체제와 동일한 OS버전으로 맞추는것이 원칙이다.

2-2. JVM
- Java 기반 응용프로그램을 구동하기 위한 인터프리터 환경으로 적용 버전은 개발 표준에 맞추어 운영환경과 동일하게 관리하는 것이 좋다.

2-3. WebServer
- 정적 웹 서비스를 수행하는 미들웨어로, 웹 브라우저 화면에서 요청하는 정적 리소스를 제공한다.
- 최신 기술 동향에서는 가볍고 빠른 Caddy, LiteSpeed등이 추가로 활용되고 있으며, 대부분 컨테이너 기반으로 배포된다.

2-4. WAS
- 동적 웹 서비스를 제공하기 위한 Tomcat, Undertow, IIS등의 미들웨어로 JSP/Servlet 어플리케이션 서버가 설치된다.
- 최근에는 Springboot, Node.js기반 MSA환경과 경량 WAS가 대세이며, 컨테이너 환경에서 운영된다.

Docker -> 프로그램들을 싹 묶어서 컨테이너로 포장해주는 프로그램
Kubernetes ->여러개의 컨테이너들을 관리하는 프로그램

2-5. DBMS
- 데이터의 저장, 관리, 검색을 위한 시스템 소프트웨어
- 최근에는 관리형 DB 서비스, NoSQL DB, NewSQL DB 사용이 급증하고 있다.

3. 개발도구 소프트웨어
3-1. 요구사항 관리도구
- 고객의 요구사항 수집, 분석, 추적, 협업을 지원한다.
- 최근에는 애자일, DevOps 연계로 협업 툴과 연동성이 높은 SaaS기반 요구사항 관리 도구가 선호된다.
- Jira, Trello
3-2. 설계/모델링 도구
- 기능을 논리적으로 설계하기 위해 통합 모델링 언어(UML), ERD, API 설계도구 등을 활용한다.
최근에는 API 다지안 툴과 클라우드 기반 협업 설계 도구 사용이 증가하고 있다.
3-3. 구현 도구
- 개발 언어에 맞는 IDE 및 편집기를 사용하여 코드를 작성하고 관리한다.
- 최근에는 Cloud IDE와 AI 개발도구 활용이 증가하는 추세이다.

3-4. 테스트 도구
- 개발된 코드의 정확성을 검증하기 위해 자동화된 테스트를 수행한다.
- 최근에는 CI/CD 파이프라인과 통합되는 테스트 툴과 보안 취약점 테스트 도구 사용이 보편화되고 있다.

4. 형상관리
- 문서, 설계 산출물, 스크립트, 빌드 결과물 등 다양한 개발 산출물을 관리대상으로 포함하여, 협업 기반 개발환경에서 필수적인 핵심 프로세스로 자리잡고 있다.
- 모든 개발 산출물이 관리 대상 협업 개발 환경에 필요한 개발 프로세스의 핵심이다.
