PL/SQL 테스트를 위한 SQL * PLUS의 활용
- PL/SQL은 변수,조건문,반복문, 예외 처리 등을 사용해서 여러 작업을 절차적으로 처리할 수 있다.
- 단순히 SQL 실행 결과만 보는게 아니라, 내부 처리 과정이 맞는지도 확인해야 한다는 것이다.

테스트 항목
실행이 되는가 : 문법 오류 없이 실행이 되는지 확인
출력이 맞는가 : 예상한 값이 출력이 되는지
데이터가 바뀌었는가 : INSERT, UPDATE, DELETE 결과를 해야 한다. -> COMMIT
오류가 없는가 : SHOW ERRORS를 통해 오류를 확인
조건별 결과가 맞는지 : 정상값, 잘못된 값, NULL값 등을 넣어 결과를 확인한다.

SQL 처리 흐름
1. 구문 분석
- 사용자 요청한 SQL문이 데이터베이스에서 처음 사용된 문장인지, 이미 사용된 문장인지를 공유 풀 영역을 검색하여 확인

※공유 풀 영역
- SQL 실행 계획을 저장하는 메모리 영역(SQL문이 보관되어 있다)

사용된적 있는 문장 -> 공유 풀에서 꺼내서 쓰면 됨(구문분석 필요X)
처음 사용되었다 -> 구문분석을 실시

구문분석
- SQL문을 데이터베이스가 이해할 수 있도록 분석하는 작업
예시
SELECT NAME
FROM MEMBER
WHERE MEMBER_ID = 1;

SELECT 문법이 올바른가
MEMBER 테이블이 존재하는가
NAME 컬럼이 존재하는가
MEMBER_ID 컬럼이 존재하는가
사용자가 MEMBER 테이블을 조회할 권한이 있는가?

- 옵티마이저는 SQL문을 가장 빠르게 데이터를 검색해줄 수 있는 실행 계획을 찾는다.

※ 옵티마이저
- SQL을 가장 효율적으로 실행할 방법을 결정하는 데이터베이스 내부 기능이다.


2. 실행
구문분석이 끝난 SQL을 실제로 실행하는 단계
- 서버 프로세스는 메모리 영역의 데이터베이스 버퍼 캐시 영역을 검색하여 해당 테이블의 데이터가 다른 사용자의 다른 SQL문에 의해 존재하는지 검색

※데이터베이스 버퍼 캐시
- 실제 테이블 데이터를 저장하는 영역

버퍼 캐시 영역에 있으면 그대로 사용

없으면 해당 테이블의 데이터파일에서 데이터를 읽고 버퍼캐시에 저장

INSERT, UPDATE, DELETE와 같은 문장이러면 데이터 버퍼 캐시 영역에서 새로운 데이터로 변경, 삭제 또는 입력을 한다.


3. 추출
실행 결과를 사용자에게 보내주는 단계
서버 프로세스는 데이터베이스 버퍼 캐시 영역에서 관련 테이블 데이터를 읽어서 사용자가 요청한 클라이언트로 보내준다.

SELECT문을 실행하는 경우에만 추출단계가 실행된다.

SQL 작성시 고려사항
- 개발자는 SQL 특성을 충분히 이해하여 SQL문을 적절히 구사할 수 있는 능력을 갖춰야 한다.
- 사용자가 SQL 작성 시, 옵티마이저가 실행 계획을 수립한 후 실행되는 일련의 과정을 이해하고 작성해야 한다.
- 구문분석 단계 시 옵티마이저가 수립한 실행 계획에 다라 엄청난 수행 속도 차이가 발생할 수 있다.
- 특정 SQL이 실행될 때 옵티마이저에 의해 수립된 실행 계획은 제어하기 어렵지만, 옵티마이저가 비상식적으로 동작한다면, 이를 추적하여 개발자가 원하는 실행 계획으로 동작될 수 있도록 조정하는 과정이 필요하다.
- 개발자는 옵티마이저가 정상적인 실행 계획을 수립할 수 있도록 종합적이고 전략적인 포인트를 SQL에 부여하여 작성하여야 한다.
- 좋은 SQL은, 추출되는 결과를 추론하여 SQL을 집합적으로 접근하여 작성하여야 한다.

- 이전에 실행된 sql문과 실행 계획을 임시 보관하므로, 동일한 SQL이 들어왔을 때 문법 검사, 오브젝트 확인, 실행 계획 수립 등의 작업을 생략한다.

버퍼캐시
- 디스크에서 한 번 읽어온 데이터를 메모리에 유지하므로, 동일한 데이터 조회 시 느린 디스크 파일 I/O를 발생시키지 않고 메모리에서 바로 데이터를 읽어와 수행 속도를 극적으로 향상시킨다.
