오류나 예외 없이 목표까지 도달하는 정상 경로를 먼저 그린다.
각 화면에서 발생할 수 있는 YES/NO 조건을 마름모로 표현
오류,실패,재시도, 이탈 경로를 점선으로 표시
빠진 화면이나 연결되지 않은 노드가 없는지 확인

작성원칙
흐름은 한방향으로 (위 -> 아래, 좌 -> 우 일관성있게)
화면당 하나의 역할 (한 박스에 여러 역할을 섞지 않는다.)
조건은 명확하게 (마름모 분기 YES/NO레이블을 반드시 표시)
예외 흐름은 구분 (점선 등으로 정상 흐름과 시각적으로 분리)
너무 세분화하지 않는다. (버튼 하나하나까지 표현하면 너무 복잡하다)

3. 유용성 개념 및 적용 원리 파악
- 유용성
사용자가 시스템을 활용하여 자신이 원하는 목표를 얼마나 효과적이고 효율적으로 달성할 수 있는지 나타내는 척도
- 유용한 시스템을 설계할 때 고려사항
사용자가 머리속에서 그리는 구조,기능,가치에 대한 이해와 실제로 설계자가 구현하려는 모형 사이의 차이를 줄이려는 노력이 필요하다.
- 사용자 모형과 개발자 모형간의 차이 발생
시스템에서 제공하는 실행 가능한 기능과 사용자가 본래 기대하는 목적이 서로 다를 때 발생
시스템이 실행된 결과가 사용자의 의도와 일치하지 않을 때 발생
3-1. 실행 차를 최소화하기 위한 UI 설계 원리
※실행 차 : 사용자가 달성하고자 하는 목적과, 실제로 수행하기 위해 시스템에 제공되는 기능 간의 간극
- 사용 의도 파악
- 행위 순서 규정
ㄴ 사용자의 행동 과정을 세분화해서 순서대로 명확하게 안내
ㄴ 한 가지 작업을 완료하는 단계는 가능한 최소화 해야 한다.
ㄴ 행위 순서는 사용자의 기존 경험과 자연스럽게 이어지도록 친숙하게 구성하는것이 좋다.
- 행위의 순서대로 실행
ㄴ 사용자가 계획한 작업 절차를 UI 상에서 직관적으로 파악할 수 있도록 제공하여, 의도한 행위를 실제로 실행하는 과정에서 혼란이나 어려움이 생기지 않도록 해야한다.
ㄴ 불필요하게 많은 상호작용으로 인해 작업 진행이 방해받지 않도록 주의해야 한다.
ㄴ사용자가 의도한 행위에 가장 적합한 입력 장치를 선택한다.
ㄴ 적절한 피드백과 취소 기능을 제공
- 기본값을 알맞게 설정해 불필요한 조작을 줄여야 한다.
3-2. 평가 차를 줄이기 위한 UI 설계 원칙
※ 평가 차 : 사용자가 사용은 했는데 피드백이 빨리 안옴
- 키 입력 결과를 사용자가 신속하게 인식할 수 있도록 설계
ㄴ 사용자가 어떤 조작을 해도 화면에 아무런 변화가 없다면, 사용자가 제대로 입력했는지 의심하게 된다. 가능한 빠르게 시스템이 반응하도록 설계 해야되고, 즉각적인 반응이 어렵더라도 지연을 최소화해야 한다.
- UI 조작으로 인해 시스템의 상태 변화를 쉽게 파악할 수 있도록 설계
ㄴ 사용자의 입력에 따라 시스템이 어떻게 달라졌는지 명확히 인식할 수 있어야 한다.
- 사용자의 의도와 시스템의 반응 결과의 일치 여부를 쉽게 확인할 수 있도록 설계
ㄴ 사용자가 기대한 결과가 실제 시스템을 통해 충족되었는지 쉽게 알 수 있어야 한다. 이를 위해 미리보기와 같은 기능을 제공하면 사용자가 예상 결과를 사전에 확인할 수 있다.

4. 스토리보드 작성 방법
스토리보드
- 서비스 구축 과정에서 디자이너와 개발자가 공통으로 참고하는 최종 산출물 정책, 프로세스, 콘텐츠 구성, 와이어 프레임, 기능 정의, 데이터베이스 연동 등 서비스 구현에 필요한 핵심 정보가 담긴다

4-1. 메뉴 구성도 작성
- 전체 메뉴의 흐름을 정리하고, 어떤 내용을 보여 줄지를 확인하여 메뉴 구성에 반영한다.
- 트리 형태 구조로 정리.
- 어떤 메뉴가 있고, 어떤 순서로 배치되며, 각 메뉴를 뭐라고 부를지 정의


4-2. 스타일 확정
- 레이아웃, 글꼴 모양과 크기, 색상, 그래픽 요소 등을 통일감 있게 유지한다.
- 기준이 없으면 화면마다 제각각이 된다.

4-3. 설계
- 실제 화면에 표현될 시각적 디자인 콘셉트를 구체화 한다.

UI 시나리오 문서 작성의 요건
- UI 시나리오 문서를 작성할 때 반드시 포함되어야 하는 내용이나 작성 기준

1. UI 시나리오 문서 목적 및 대상 사용자
목적 : 핵심 사용자 흐름과 과업 중심 상호작용을 명확하게 기술하여, 실제 서비스 구현 시 혼선이 발생하지 않도록 한다.
대상 : 화면 설계에 참여하는 모든 인원(기획자, 디자이너, 퍼블리셔, 개발자 등)

2. 구성요소
- UI 시나리오 문서에는 
개요 : 사용자 목표, 시나리오 목적, 기본 전제 조건 제시
사용 흐름 : 단계별 화면 이동, 상태 변화, 화면 전환 로직
UI 요소 구성 : 아이디어 도출 및 설계 방향성 탐색
상호작용 및 입력 로직 : 버튼 클릭, 입력 필드, 오류 상황 대응, 피드백 흐름 정의
접근성 고려사항 : 웹 접근성 기준 준수, 키보드 제어, 명확한 라벨과 판독성 확보
예외 및 오류 시나리오 : 오류 발생 시 흐름, 복구 절차, 사용자 안내 메시지 설계
적용 수준 : 필수, 권장, 우수 수준으로 구분하여 단계적 적용 계획 수립이 포함되어야 한다.

3. 작성원칙
- 일관된 표현의 사용
디자인 원칙, 스타일 가이드에 다른 컴포넌트 명칭과 용어를 일관되게 사용하기

버튼 (확인, 완료 저장하기) -> 저장
사용자 (회원, 유저, 사용자) -> 사용자
로그인 ID (계정, ID, 아이디) -> 아이디
오류 (에러, 오류, 실패) ->오류

- 사용자 중심

- 현장 적용 가능
