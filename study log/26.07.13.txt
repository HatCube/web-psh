병원 진료 시스템의 구현
각 클래스는 필요한 객체를 직접 생성하지 않고, 스프링 컨테이너가 생성한 Bean을 생성자를 통해 전달받는다.

의존관계는 다음과 같다.

Hospital -> Doctor -> Patient

Hospital은 Doctor를 사용한다
Doctor는 Patient를 사용한다
Patient는 다른 객체에 의존하지 않는다.

Patient클래스
ㄴ private String name = "홍길동"
ㄴ public void checkInformation() -> "환자 홍길동의 정보를 확인합니다."

Doctor클래스
ㄴ Patient 객체를 생성자를통해 주입받는다.
ㄴ public void treat() : 
- "의사가 환자 진료를 시작합니다" 출력
- Patient 객체의 checkInformation()메서드 호출
- "환자 진료가 완료되었습니다" 출력

Hospital 클래스
ㄴ Doctor 객체를 생성자를 통해 주입받는다.
ㄴ public void startMedicalService()
- "병원 진료 업무를 시작합니다"
- Doctor 객체의 treat()메서드 호출
- "병원 진료 업무를 종료합니다."

백엔드 서비스 아키텍처
레이어드 아키텍처 패턴
- 어플리케이션을 역할별 계층으로 나누어 구성하는 설계 방식
사용자 요청 처리 -> 업무 로직 처리 계층 -> 데이터베이스 접근
- 각 계층이 담당하는 역할을 명확하게 나누는 것이 핵심이다.

계층 구조
Presentation Layer(표현 계층) : Controller
- HTTP 요청 받기
- 요청 데이터 검증
- Service 호출
- HTTP응답 반환

Business Layer(비즈니스 계층) : Service
- 실제 업무 로직을 처리하는 계층이다

Persistence Layer(영속 계층) : Repository
- 데이터베이스에 접근하는 계층
- 데이터 저장, 조회,수정,삭제

데이터베이스 서버

엔티티 VS DTO

Entity : 데이터베이스 테이블을 표현하는 객체
DTO : 계층간 데이터를 주고받기 위한 객체

Entity만 사용했을 때 문제점
- 불필요한 정보까지 주고받는다 (ex: 비밀번호)

@RestController
@Controller + @ResponseBody
해당 클래스를 컨트롤러라고 스프링에게 알려준다.
요청을 받고 처리를 거친 후 응답을 통해 화면이 아닌 데이터를 돌려준다.

@RequestMapping("리소스")
- HTTP 요청(URL)을 특정 클래스나 메서드와 연결하는 어노테이션이다

http://localhost:8080/test -> 스프링은 "이 요청을 어떤 메서드가 처리해야하지?"

@RequestMapping("test")
public class TestController { ... }

/test로 시작하는 요청은 모두 TestController가 처리한다.

@GetMapping("경로")
Get으로 들어오는 요청을 받아서 처리하겠다.

@PathVariable
경로의 일부를 변수로 만들어서 매개변수에 데이터를 넣어주는 어노테이션
경로 변수와 매개변수의 이름이 같으면 생략 가능
- 리소스 자체를 식별할 때 사용
/users/10 -> 10번 회원 -> select * from member where id = 10;
/products/5 -> 5번 상품
/orders/20 -> 20번 주문

@RequestParam
검색 조건이나 옵션을 전달할 때 사용
데이터를 쿼리스트링 방식(키=값)으로 받는다.

프로토콜://도메인/경로(리소스)?쿼리스트링
http://localhost:8080/users/10/orders/5

쿼리스트링
겁색 조건이나 옵션을 전달하는 값
?이름=값&이름=값&이름=값

기본값 설정
@RequestParam의 파라미터가 선택적일 때, 기본값을 설정할 수 있다.
클라이언트가 해당 파라미터를 생략했을 때 기본값이 저경된다.

REST API
- REST(Representitaional State Transfer)아키텍처를 기반으로 하는 API이다.
- 클라이언트와 서버 간의 상호작용을 HTTP 프로토콜을 사용해 설계한 웹 서비스의 인터페이스로, 데이터를 쉽게 주고받을 수 있게 해준다.
- 웹 어플리케이션, 모바일 앱, 마이크로서비스 등 다양한 시스템 간의 통신을 위한 표준으로 널리 사용된다.
- REST 아키텍처 스타일은 6가지 제약조건으로 구성된다.
- 이 가이드를 따르는 API를 RESTful API라고 한다.

REST API의 주요 개념
클라이언트 - 서버
리소스를 관리하는 서버가 존재하고, 다수의 클라이언트가 리소스를 소비하기 위해 네트워크를 통해 서버에 접근하는 구조

상태가 없는(stateless)
클라이언트가 서버에 요청할 때, 이전 요청의 영향을 받지 않는다.
/login을 통해서 로그인하고 /page로 이동할 때 서버는 로그인 사실을 모른다.
그렇기 때문에 로그인 상태를 유지하려면 다른 요청을 보낼때마다 로그인 했다는 증명을 같이 보내야한다.

http 메서드 사용의 일관성
GET : 리소스를 조회할 때 사용
POST : 새로운 리소스를 생성할 때 사용
PUT : 기존 리소스를 업데이트 할 때 사용
DELETE : 리소스를 삭제할 때 사용

GET /users  -> 모든 사용자 조회
GET /users/{id} -> 특정 사용자 조회
POST /users -> 새로운 사용자 생성
PUT /users/{id} -> 사용자 정보 업데이트
DELETE /users/{id} -> 사용자 정보 삭제

GET /posts -> 모든 게시물 조회
GET /posts/{id} -> 특정 게시물 조회
POST /posts -> 새로운 게시물 생성
PUT /posts/{id} -> 게시물 업데이트
DELETE /posts/{id} -> 게시물 삭제

URI 구조의 일관성
- 경로를 일관되게 설계하는 것도 중요하다
- 리소스의 이름은 복수형으로 일관되게 사용하고, 리소스간 관계를 표현할 때도 규칙적으로 작성해야한다.

GET /users/{userId}/posts
POST /users/{userId}/posts
GET /users/{userId}/posts/{postId}
